- **Agent SDK Phase 1 shipped (issues 933–937 + riders 960/962/963/964; PRs 955–959, 961, 965, 967, 968)** — persistence + per-call cancellation; the load-bearing facts:
  - **`RunStore` lives in `agent/runstore.go`** (A6 corollary: it traffics in `agent.Message`/`Event`, so root `stores/` would force root→agent). gRPC-style req/resp per `stores/STORAGE_SEAMS.md`; unknown RunID is app-state (`Found=false`), never an error. **Atomicity contract on the interface**: explicit-ID CreateRun and ForkRun are all-or-nothing; `NewRunID` is a retry idempotency key. Backends: `agent/store/redis` (fork = one atomic Lua script, meta-write last; cluster caveat doc'd) and `agent/store/gorm` (fork = one transaction; sqlite + Postgres, `just testpg` boots a container on **5435** — events' gorm store owns 5434). Rationale for "native atomicity primitive per backend, never hand-rolled claim-last ordering" is doc'd on both ForkRuns.
  - **Runs are flat copy-on-fork lists** — messages carry no IDs or parent pointers; `Run.ParentID`/`ForkPoint` are lineage metadata only, nothing reconstructs history by walking. `ForkPoint` is a **message count, not a timestamp** (renamed from the issue's ForkedAt for exactly that ambiguity). `ForkRunRequest.AtMessage` forks from an earlier point (checkpoint/rewind's conversation half); **partial forks copy NO events** (audit stream isn't sliceable by message index). Stored blobs being immutable is what will make issue-966 offloading fork-free.
  - **Stores stamp `Message.Timestamp` on AppendMessages** (zero gets the store clock, caller-set wins — same caller-preserves rule as trace context). Field uses **`omitzero`, not `omitempty`** (omitempty does NOT omit a zero `time.Time`); providers never map it into request bodies. The stamp lives inside the stored JSON body — no schema column, forks copy it free.
  - **Host wiring** (`agent/host/persistence.go`): `WithRunStore` + `AttachRun` (create-or-resume) / `Resume` (must-exist) / `Fork`; persists at the `RunTurn` append site; `PersistingEmit` buffers the event stream, one `AppendEvents` per turn. Failed turns persist nothing; persistence failures render a warning, never fail the turn. agentchat: `--session-store memory|sqlite://path.db|redis://host:port|postgres://user:pass@host:port/db` + `--session <id>`; sqlite gives restart-surviving sessions with no server.
  - **Per-tool-call cancellation**: `Runner.RunTurn(ctx, TurnRequest)` with `Control <-chan Control`; `Run` is a shim. `Control{}` (empty CallID) cancels ALL in-flight calls in one send; `Control{CallID}` targets one; the cancelled call feeds `"cancelled by user"` back and **the turn continues**; cancelling RunTurn's own ctx still aborts. **Gotcha that bit**: cancellation must be checked on all three outcome shapes — transports error, but `FuncSource` converts handler `ctx.Err()` into an `IsError` result, and a success can race the cancel (user cancel wins). `EventToolCancelled` carries `Reason` (not `Error`) so error-keyed eval scorers don't miscount interrupts. Control is the extensible steering envelope (future verbs = additive `Kind`); mid-turn *content* ("/btw") is deliberately NOT a Control — it routes through injection so it lands in history.
  - **Event-log dedup shipped** (was a Phase 1 follow-up): tool-result payloads via offloading (966) and the `turn-end` `TurnResult.Messages` duplication via `PersistingEmit` redaction (973) — see the Phase 2 bullet.

- **Agent SDK Phase 2 opener shipped — tool-result offloading (issues 966/971/972/973/977; PRs 970/975/976/978/981)**. The just-in-time-context layer: lossless, pay-on-lookup, complementary to (not a replacement for) compaction. Design map: `docs/AGENT_MEMORY_FLOW.md`. Load-bearing facts:
  - **`OffloadingSource` wraps a `ToolSource` like `FilterSource`** (`agent/offloading_source.go`): an over-threshold *successful* result is stored and replaced by a **stub** (a normal `ToolResult` whose text is a preview + ref), so the RoleTool message, `tool-end` event, and persisted log all carry the stub — the log stays faithful to what the model saw, no Runner change. Wrap the **aggregate** MultiSource (host does this when `Config.Offload != nil`) so one `read_tool_result` + one store cover every server. **`IsError` stays inline** (truncating an error is worse); per-tool threshold `0` pins a tool inline. Offload size is measured on `toolResultText` (**text only** — a big base64 blob wouldn't trip it yet; issue 979).
  - **`ToolResultStore` seam** (`agent/toolresultstore.go`, A6: it traffics in `core.ToolResult`): `Put`/`Get`, in-memory default. **Retention is deliberately NOT in the interface** — the graceful unknown-ref read (`read_tool_result` → "no longer available", `IsError:false`) is what makes any backend eviction safe, so redis uses native TTL and gorm a caller-driven `PruneExpired`. Durable backends live in the existing `agent/store/redis` + `agent/store/gorm` siblings (heavy deps); the **dep-free `FileToolResultStore` lives in `agent/` itself** (stdlib only — one JSON file per ref, temp-then-rename, injective ref→filename escaping), the natural no-server store for a local/coding agent (blobs are files the agent can read directly). gorm backend supports **bring-your-own-table** (`WithToolResultTableName`, resolved via `db.Table`).
  - **read_tool_result** takes `{ref, offset?, limit?, pattern?}` — a char window or a regex grep *into* the blob (query, don't page). Text-only today; binary retrieval is issue 979.
  - agentchat: `--offload-threshold <bytes>` enables it, `--offload-dir <path>` routes blobs to a `FileToolResultStore` (no DB), else reuses `--session-store` so one spec backs both runs and blobs on one connection.
  - **Deferred**: binary offloading (979 — size trigger + non-text retrieval), streaming/handle-based results for 100s-of-MB via MCP resource links + a ranged store (980 — `Put` is NOT the bottleneck; upstream materialization is). **Evals-first for the memory tracks**: LongMemEval-derived cases (974) gate compaction before any lossy work.

- **Agent SDK Phase 3 (multi-agent composition) COMPLETE — epic 927 closed (issues 941/942/943; PRs 1028/1029/1030 + example 1034 + doc 1037)**. Design frame: `docs/AGENT_COMPOSITION.md`. Load-bearing facts:
  - **The two-axis model**: multi-agent is NOT a new engine — it wraps the same stateless `Runner`. **Context in = injection** (server events + memory recall + handoff context, one seam). **Control = tools + signals** (936 cancel down · signals up · runner-control meta-tools · opt-in interruptible turn). **Observability = the emit stream** (`SubAgentEvent` nests it). The Runner stays a deterministic **fan-out-then-join** pure function; composition wraps it (the interruptible turn is the one opt-in exception, so resume/fork/eval/compaction stay composable).
  - **`AgentSource`** (`agent/agent_source.go`, A6): agent-as-tool — a child `*Runner` exposed as a `ToolSource`; `Call` runs the child over its OWN isolated slice (seeded with `{task}`) and returns its final text. Isolation is structural, so the Runner never changes; **supervision falls out via a `MultiSource` of `AgentSource`s** (existing aggregation/collision/Selector). Guards: per-source `MaxDepth` (ctx-threaded, default 3) + optional ctx-threaded aggregate `WithAgentCallBudget` across the whole tree. A refused/failed child is an `IsError` result (parent turn continues); only an unknown name is a dispatch error. **The LLM decides to call a sub-agent** (by the tool's name+description), not the parent code; the Runner is a dumb dispatcher and **parallelizes tool calls in one turn** (so N sub-agents in one turn run concurrently, ordered results).
  - **`SubAgentEvent{Scope, Depth, Event}`** (942): the child's turn-**lifecycle** emit stream (`turn-begin`/`tool-begin`/`text-delta`/`turn-end`) forwarded to the parent's surface via `AgentSourceConfig.OnEvent`, scoped/depth on the **envelope** so `Event` stays wire-flat (A2). Scope is ctx-threaded → nested sub-agents compose the path (`outer/inner`). **NOT** MCP domain-event subscriptions — a sub-agent's `subscribe_events` injects into its OWN context, never the parent's; the parent sees the child's *activity*, not its *inputs*.
  - **`Team`** (`agent/team.go`, 943): **handoff** = transfer control, don't return — the one mode that doesn't fit agent-as-tool. `NewTeam(TeamConfig{Members, Start, MaxHandoffs, OnHandoff})`; each `TeamMember{Name, Config, HandoffTo}` gets `transfer_to_<name>` tools (only for allowed targets; terminal agents get none). `Run` loops: active agent runs → if a `transfer_to` fired (per-Run ctx signal), swap the active Runner + continue over the SAME shared history → else final answer. **Static membership** (declared at construction; dynamic model-driven composition is 1038). Ping-pong bounded by `MaxHandoffs` (`ErrMaxHandoffs`). Handoff = **inject-context + schedule**; the shared-thread swap is one impl, per-agent-context + injection (actor form) is the general one.
  - **Multi-model folds naturally**: each `Runner`/`AgentSource`/`TeamMember` carries its own `Provider`, so a heavy supervisor delegating to cheap sub-agents = different providers per child — no special machinery. Within one conversation: `ConnectionRegistry` + `providerSwitch` (`/provider`), `FailoverProvider` (primary/backup), and the deferred routing policy (991, per-turn/per-role selection).

- **`ext/skills` is data-only, enforced (issue 839 / PRs 840, 842)**: a skill is data delivered over resource primitives, never staged to disk, never executed. `TestNoCodeExecutionSurface` (`ext/skills/noexec_test.go`) is an AST scan that fails the build if the package imports `os/exec` / `plugin` or calls a filesystem-write / process-spawn function — including the lower-level spawn primitives (`os.StartProcess`, `syscall.Exec` / `ForkExec` / `StartProcess`) so it can't be bypassed by dropping below `os/exec`. **A loader that downloads + persists remote skills to the serving folder would trip this by design** — such a loader belongs OUTSIDE core `ext/skills` (opt-in, e.g. `ext/skills/loader`), with the archive traversal/symlink/bomb guards re-applied at the write boundary and integrity verification on fetch. `FetchGitHubArchive` already loads from GitHub without disk staging (fetch → in-memory `SourceFS` → serve), so that stays the default. `adversarial_test.go` maps `ext/skills` case-by-case onto the WG's `dangerous-skills-mcp` corpus (non-archive slice: digest mismatch, frontmatter, directory-read traversal) and asserts the one known gap — non-archive index digest covers `SKILL.md` only, so supporting files are unpinned (whole-skill-digest, still open upstream).
- **Two-tier skills loading — catalog mode + `load_skill` (issue 910 / PR 1073)**: eager full-body injection bloats the system prompt at scale (50 skills × ~100 tok, resent every call), so progressive disclosure. The catalog decision is **per-SERVER, not per-skill/per-set** — `ServerConfig.SkillsMode` (`"eager"|"catalog"|""`) marks a whole skills server; `""` auto-selects by **skill-md count** (`defaultCatalogThreshold = 10`: ≥10 → catalog, else eager). `skills:false` stays the off switch; `skillsAllow` (per-skill hard filter) is a separate deferred item. **eager** = `skills.InstructionsBlock` (full bodies) into the prompt; **catalog** = `skills.CatalogBlock(idx)` (one `- name: desc` line per skill-md entry, ~a tenth of the tokens) + collect entries. **One** `load_skill(name)` FuncSource tool spans ALL catalog-mode servers (not per-server/per-skill): find entry by name → `ReadAndVerify(url, digest)` → body as a **tool result** (into tool-history, NOT re-injected into the prompt — swinging the prompt is prompt-cache-hostile); laziness never bypasses digest verification; unknown name = app-state, not an error. Chose per-server mode over the connect-time `SkillsPolicy` hook (embedder-first; can coexist). Byte-threshold + body caching + the policy hook are deferred.
