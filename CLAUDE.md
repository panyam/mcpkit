# CLAUDE.md — MCPKit

Go library for building production-grade MCP servers and clients.

## Quick Commands

```bash
make test              # Core tests (core/server/client/testutil)
make test-auth         # ext/auth sub-module
make test-ui           # ext/ui sub-module
make test-e2e          # E2E tests (auth + apps)
make testconf          # MCP conformance suite (needs Node.js)
make testconfauth      # Auth conformance (client OAuth)
make testconf-tasks    # Tasks v1 conformance (27 scenarios, local)
make testconf-tasks-v2 # SEP-2663 — upstream @ MCPCONFORMANCE_TASKS_V2_PATH + mcpkit-local stricter sentinel
make testconf-mrtr     # SEP-2322 — upstream @ MCPCONFORMANCE_MRTR_PATH + mcpkit-local stricter sentinel
make testconf-list-ttl # SEP-2549 — fork @ MCPCONFORMANCE_LIST_TTL_PATH (5 checks, 3 fixtures)
make testconf-file-inputs # SEP-2356 — fork @ MCPCONFORMANCE_FILE_INPUTS_PATH (7 checks)
make testconf-auth-server # MCP authz 2025-11-25 — fork @ MCPCONFORMANCE_AUTH_PATH (6 scenarios, 23 checks: 18 active, 5 SKIPPED gaps for unsupported features)
make testconf-stateless # SEP-2575 — upstream @ MCPCONFORMANCE_STATELESS_PATH; drives examples/stateless fixture (25 pass / 1 known upstream-test fail)
make testconf-skills   # SEP-2640 — fork @ MCPCONFORMANCE_SKILLS_PATH; drives examples/skills fixture
make testconf-upstream-audit # Audit mcpkit against modelcontextprotocol/conformance@main → conformance/UPSTREAM_AUDIT.md (informational scenario-level pass/fail; runs in the conformance umbrella but exits 0 by design)
make refresh-conformance # Regenerate CONFORMANCE.md from upstream tier-check + traceability (issue #498). Driver: scripts/refresh-conformance.sh, renderer: tools/conformance-report.
make check-conformance-stale # CI gate — refresh + git diff --exit-code CONFORMANCE.md (issue #498). Wired into .github/workflows/test.yml on every PR.
make testall           # Everything (9 stages, 21 sub-stages) + Keycloak + HTML report
                       # stage 5a/5b cover ext/otel + examples/otel/stdout (issue 634 wiring)
                       # stage 7d covers experimental/ext/events/stores/redis (issue 634 wiring)
make audit             # govulncheck + gosec + gitleaks + race
make tag-push V=vX.Y.Z # Tag root + all sub-modules and push
```

## Sub-Modules

`ext/auth/`, `ext/tasks/`, `ext/ui/`, `ext/otel/`, `experimental/ext/protogen/`, `docs/site/` have separate `go.mod` — `make test` does NOT cover them. Run `make tidy-all` after touching `core/` imports. `docs/site/` is the GitHub Pages renderer (issue 508); it is a tool, not a library, and is excluded from `SUB_MODS_TO_TAG`.

## Constraints

Project-wide: `CONSTRAINTS.md`. Per-package: `core/CONSTRAINTS.md`, `server/CONSTRAINTS.md`, `client/CONSTRAINTS.md`.

## Gotchas

- **`GH_TOKEN="$GH_PERSONAL_TOKEN"`**: EMU account can't access personal repos
- **Sub-module go.sum drift**: New `core/` imports break sub-modules until `make tidy-all`
- **Server requires initialization**: Direct `srv.Dispatch()` in tests fails — use httptest + client
- **AppHost lifecycle**: `Client.Connect()` before `AppHost.Start()`. `AppHost.Close()` only closes bridge.
- **Background goroutines**: Use `core.DetachForBackground(ctx)` (not `context.WithoutCancel`) — replaces dead POST-scoped requestFunc/notifyFunc with the session-level persistent push.
- **CORS for browser clients**: MCP servers need `Mcp-Session-Id` in both Allow-Headers and Expose-Headers, plus `DELETE` in allowed methods. Use `servicekit/middleware.CORS()` with options.
- **Demokit non-interactive + browser steps**: Steps that open a browser and expect user action will fail in `--non-interactive` mode. Interactive mode is the primary path.
- **SEP-2549 `ttlMs` uses `*int` + omitempty**: the merged spec treats an absent `ttlMs` the same as `0` (both "immediately stale"), so the field is two-state client-side. mcpkit still types it `*int` so a server can emit an explicit `ttlMs: 0` distinct from omitting the field — plain `int` + omitempty cannot. `cacheScope` is a plain `string` (omitempty); absent defaults to `"public"` client-side. Field was renamed from `ttl` (seconds) during the spec's final review. See `docs/LIST_TTL_MIGRATION.md`.
- **Conformance suites are brand-neutral**: `conformance/*/` assert what the spec says, not what mcpkit does. mcpkit-specific behavior (e.g., echoing SEP-2243 routing headers on responses) belongs in `server/*_test.go`, not in conformance scenarios. The conformance suites are the marketing — keep them framed as "what any server must do."
- **`MCPCONFORMANCE_*_PATH` (per-suite, defined in `conformance/Makefile`)**: most `testconf-*` targets point at a worktree of the [`panyam/mcpconformance`](https://github.com/panyam/mcpconformance) fork because some SEPs still live on draft branches. Tasks v2, MRTR, and stateless now point at upstream [`modelcontextprotocol/conformance`](https://github.com/modelcontextprotocol/conformance) `main` directly, since those scenarios merged. Defaults (resolved relative to `conformance/Makefile`): `MCPCONFORMANCE_TASKS_V2_PATH` and `MCPCONFORMANCE_MRTR_PATH` → `../conf-upstream-main` (`modelcontextprotocol/conformance@main`, merged upstream); `MCPCONFORMANCE_FILE_INPUTS_PATH`, `MCPCONFORMANCE_LIST_TTL_PATH`, and `MCPCONFORMANCE_AUTH_PATH` → `../conf-pending` (`pending`); `MCPCONFORMANCE_SKILLS_PATH` → `../conf-main` (main, PR 330 in the fork). `MCPCONFORMANCE_BASE_PATH` (used by `testconf-upstream-audit` and the stateless suite) also points at `../conf-upstream-main`; the audit reads the canonical scenario set, so any fork drift would defeat its purpose. Override per-invocation when a SEP splits to its own branch waiting upstream approval. Each target fail-fasts with a remediation message if its path is missing.
- **Upstream conformance audit (`conformance/UPSTREAM_AUDIT.md`)**: regenerated by `make testconf-upstream-audit`. Snapshot of mcpkit graded against every scenario upstream currently ships, grouped by SEP. Runs in the conformance umbrella (`make -C conformance test`) and exits 0 regardless of scenario verdicts — the audit is informational, not a gate, but its inclusion in the umbrella keeps the committed snapshot fresh. Report header carries no wall-clock timestamp; the file only diffs when upstream advances, mcpkit advances, or a scenario flips status. Use the report to prioritize implementation work; each `fail` / `harness-gap` row is a follow-up. The `also-covered-by-fork` tag is hand-maintained in `scripts/conformance-audit-report.ts` — update the `FORK_OVERLAP` map when SEP-fork targets gain or lose coverage.
- **Fixture audit (`conformance/FIXTURE_AUDIT.md`)**: companion artifact to `UPSTREAM_AUDIT.md`. Where `UPSTREAM_AUDIT.md` grades mcpkit against the upstream test set, `FIXTURE_AUDIT.md` grades the fixtures/drivers/scripts that *back* those tests — verifying we pass the suite using the same API paths real mcpkit users are taught to use, not undocumented backdoors. Point-in-time snapshot (not auto-regenerated); re-audit when a new fixture lands under `examples/`, when a new typed helper obsoletes a JUSTIFIED entry, or before a release. See the doc's "Re-audit triggers" section.
- **SEP-2575 wire mode defaults are asymmetric**: server `stateless.DefaultMode = stateless.ModeDual` (additive on upgrade — every existing server gains the stateless wire on one URL); client `client.DefaultClientMode = client.ClientModeLegacyOnly` (conservative — `Adaptive` would have silently broken 11 pre-existing client tests that assume the legacy initialize handshake). Override per-deployment via constructor option (`server.WithStatelessMode(...)` / `client.WithClientMode(...)`), env var (`MCPKIT_STATELESS_MODE` / `MCPKIT_CLIENT_MODE`), or `init()` flip of the package var. The shipping client default may flip to `Adaptive` in a future major release; doc-block calls out the migration path.
- **`HandleStore[T]` is opt-in scaffolding, not a SEP-2567 contract**: SEP-2567 is design guidance only — no wire contract, no upstream conformance. Any storage a tool handler can call (Redis, SQL, sync.Map, custom RPC) satisfies the pattern. `server.HandleStore[T]` ships the typed in-memory default + interface seam; use it, replace it, or skip it — all three are equally SEP-2567-compliant. See `docs/SEP_2567_HANDLES.md`.
- **`ctx.Sample`/`ctx.Elicit` are stateless-wire forbidden** (server-initiated push doesn't exist there). Tool handlers route through MRTR via `core.NewSamplingInputRequest` / `core.NewElicitationInputRequest` + matching decoders. The legacy push API errors out with `ErrNoRequestFunc` on stateless requests by construction. Godocs on Sample/Elicit spell out the migration with worked examples.
- **SEP-2577 deprecation pass (issue 316)**: Roots, Sampling, and Logging surfaces all carry `// Deprecated:` blocks pointing at `docs/SEP_2577_DEPRECATIONS.md`. v0.3.x keeps every API working — *no behavior change*; only `staticcheck SA1019` warnings fire at call sites. Removal target is v0.4. Affected public symbols: `core.RootsListResult`, `IsPathAllowed`, `AllowedRoots`, `SetAllowedRoots`, `BaseContext.IsPathAllowed/AllowedRoots`, `NewListRootsInputRequest`, `DecodeListRootsInputResponse`, `server.WithAllowedRoots`, `server.WithRootsFetchTimeout`, `client.RootsHandler`/`WithRootsHandler`/`Client.NotifyRootsChanged`; `core.SamplingMessage`, `ModelHint`, `ModelPreferences`, `SamplingMeta`, `CreateMessageRequest`, `CreateMessageResult`, `ErrSamplingNotSupported`, `Sample`, `BaseContext.Sample`, `NewSamplingInputRequest`, `DecodeSamplingInputResponse`, `server.TaskContext.TaskSample`, `client.SamplingHandler`/`WithSamplingHandler`; `core.LogLevel` + constants, `LogMessage`, `EmitLog`, `BaseContext.EmitLog`, `ParseLogLevel`, `MCPLogHandler` + `MCPLogHandlerOptions` + `NewMCPLogHandler` + `SlogToMCPLevel` + `MCPToSlogLevel`.
- **Handler return ABI is sealed-interface (issue #486 / PR #487)**: `ToolHandler` returns `(core.ToolResponse, error)`; `PromptHandler` returns `(core.PromptResponse, error)`. Concrete `ToolResponse` variants are `ToolResult` (sync), `InputRequiredResult` (MRTR), `CreateTaskResult` (SEP-2663 task envelope), `GoAsyncResult` (in-process spawn signal). `core.ToolResult` no longer carries `IsInputRequired`/`InputRequests`/`GoAsync` sentinel fields — they live on dedicated variant types. `ctx.RequestInput` returns `(core.InputRequiredResult, error)`. Handler bodies usually don't change: `return core.ToolResult{...}, nil` still compiles. Use `core.TypedTool[X, core.ToolResponse]` for handlers that return polymorphic variants. Migration recipe: `docs/HANDLER_RETURNS_MIGRATION.md`.
- **Stateless-wire dispatch parity with the legacy path** (`server/stateless_backend.go::callToolForStateless`): the stateless `tools/call` path must mirror every pre-handler step `Dispatcher.handleToolsCall` runs, or features silently no-op on the SEP-2575 wire. Three landed: MRTR (`0202c926` — decodes the full envelope: `inputResponses` + `requestState`, verifies/merges/re-mints), SEP-2356 file-input validation (PR 834 — `validateFileInputArgs` before the handler), and on the client side the SEP-2243 `Mcp-Name` routing header for task ops (PR 832 — `core.DeriveMcpName` task cases + map params). `testconf-mrtr` runs both wires green. **The recurring failure mode**: a dispatch feature added to the legacy path but not to `callToolForStateless` works on legacy and silently fails on stateless; conformance often misses it because task/file-input suites run legacy while the stateless suite uses the cart fixture. `make verify-dual` (issue 478) drives example walkthroughs over both wires to catch this class; see `examples/DUAL_MODE_AUDIT.md`.
- **Stateless dispatch surfaces typed middleware errors (PR 816, issue 815)**: a `server.Middleware` that short-circuits with `*core.AuthError` (e.g. SEP-2350 scope-challenge / step-up auth) now reaches the transport's shared `writeAuthError` on the SEP-2575 wire, emitting HTTP 403 + `WWW-Authenticate` — previously folded into a generic `-32603`/HTTP 200, losing the status and header. Mechanism: `stateless.Backend.InvokeWithMiddleware` returns `(*core.Response, error, bool)` and `stateless.Dispatcher.Dispatch` returns `(*core.Response, error)` (was single-value); the three middleware-bearing branches (tools/call, prompts/get, custom-method default) forward the raw err, the rest wrap with `nil`. `handleStatelessPost` + `handleStatelessPostSSE` call `writeAuthError` on a non-nil dispatch error (SSE path is safe — middleware short-circuits before any frame is written, so `sseStarted` is always false). **Behavior-parity note**: this routes ALL stateless middleware errors through `writeAuthError`, so a *non*-`AuthError` middleware error now maps to HTTP 401 (matching the legacy wire) instead of `-32603`/HTTP 200. **Still uninstrumented**: stateless `resources/read` / `completion/complete` don't traverse the middleware chain at all — a separate, broader gap (auth middleware can't gate them on the stateless wire yet).
- **Lazy `OAuthTokenSource` scope acquisition (PR 820, issue 818)**: `ext/auth.OAuthTokenSource.Token()` no longer pre-acquires eagerly. Until a server 401/403 challenge selects the scope (or the caller pins explicit `Scopes`), `Token()` returns `core.ErrNoTokenYet` and runs no OAuth flow. The client transport's `DoWithAuthRetry.SetAuth` treats that sentinel as *skip-header* — sends the request unauthenticated so the server's `WWW-Authenticate scope=` drives selection per RFC 6750 §3.1, then `OnUnauthorized` arms the source via `TokenForScopes`. **`OnUnauthorized` now routes EVERY scope-aware 401 through `TokenForScopes`** (dropped the `len(scopes) > 0` guard PR 819 added) — a scope-less 401 is an empty merge that still flips the source out of its deferred state, so the retry acquires via the PRM `scopes_supported` fallback instead of looping on `ErrNoTokenYet`. Mechanism: `OAuthTokenSource.armed bool` gate in `Token()`; `TokenForScopes` sets it; `Invalidate` deliberately leaves it set so a step-up retry doesn't re-defer. Discovery no longer pins scope — `MCPAuthInfo.Scopes` is **removed** (probe `WWW-Authenticate` scope capture + the Step-5 PRM-fallback assignment both dropped); acquisition reads the catalog from `info.PRM.ScopesSupported` (so does `ClientCredentialsTokenSource`, which stays eager). Fixes the eager-PRM bug where a per-method-gated server (`initialize` open, `tools/list` needs `mcp:basic`) saw the client request the broad PRM `mcp:profile` first. **Behavior change**: standalone `Token()` callers get `ErrNoTokenYet` until armed or until they set explicit `Scopes` (documented on the type + `Token`). conf-auth fork `auth/scope-step-up` → 24/24 (`scope-step-up-initial` was a WARNING). The lazy path depends on PR 819's 401 scope-retry wiring (merged).
- **apps/compat Playwright baselines are Docker-pinned to Linux** (PR 534, 537, 538): `make test-apps-playwright` runs upstream's `ext-apps` Playwright suite against a mcpkit-Go drop-in under `examples/apps/compat/<name>/`. One canonical baseline per fixture (no platform suffix), pinned to `mcr.microsoft.com/playwright:v1.57.0-noble` — the same image upstream uses for `test:e2e:docker`. Regenerate via `make test-apps-playwright-docker` (`DOCKER=1`). Native mode is for fast local `loads app UI` iteration; the `screenshot matches golden` test will fail against the Linux baseline on macOS/Windows hosts — intentional, use DOCKER=1 for the visual gate. DOCKER mode also runs a **strict** `tools/list` parity check against upstream's TypeScript reference server on a side port — any divergence fails the build. The diff filters `$schema` (different SDKs emit different draft URLs) and `additionalProperties` (mcpkit's permissive default per `core/schema.go`); everything else is enforced. Wrapper env vars (`HARNESS_PORT` / `SANDBOX_PORT` / `FIXTURE_PORT` / `UPSTREAM_PORT` / `EXT_APPS_DIR` / `DOCKER` / `SKIP_DRIFT_CHECK`) and the drop-in pattern: see `examples/apps/compat/README.md`. Per-fixture committed PNGs (not upstream's tree) because basic-host renders one dropdown entry per server, and compat runs spin up only 1 server vs upstream CI's 25.
- **Two-TP pattern for ext/auth (PR 694 + PR 699)**: ext/auth's `JWTConfig` carries TWO distinct TracerProvider fields. `TracerProvider mcpcore.TracerProvider` gates ext/auth's OWN spans (`auth.jwks_lookup` sub-span + `mcp.auth.*` attributes on the active dispatch span via `core.SpanFromContext`). `OneauthTracerProvider trace.TracerProvider` (OTel SDK type, not mcpkit's abstraction) gets threaded via `keys.WithTracerProvider` to enable oneauth's internal spans (`oneauth.jwks.*` / `oneauth.signature_verify`). The two fields are deliberately distinct because oneauth's API takes the OTel SDK type directly — unwrapping inside ext/auth would couple it to ext/otel. Adopters bridge the two via `commonotel.UnderlyingOTelTP(tp)` which type-asserts to `*mcpotel.Provider` and returns the underlying OTel TP (nil for Noop). Requires oneauth ≥ v0.1.14 (current pin v0.1.19 — bumped to pick up `LoginRequest.ClientSecret` + the AS-discovery fixes the whole-enchilada demo needs). When bumping oneauth, **sweep all 8 modules in lock-step** (cmd/testclient, ext/auth, tests/keycloak, tests/e2e, examples/auth, examples/fine-grained-auth, examples/events/discord, examples/events/telegram) + `make tidy-all` — leaving even one behind fails CI's test-auth job via tests/e2e MVS resolution. ext/auth depends on the core abstraction only for its own spans; no compile-time dep on ext/otel.
- **MRTR multi-round trace stitching (PR 732, issue 682)**: SEP-2322 MRTR splits one logical operation across N `tools/call` dispatches; without intervention each round = its own W3C trace. PR 732 adds `_meta.io.modelcontextprotocol/tracelink` (vendor namespaced because W3C doesn't define `tracelink`) carrying round-1's traceparent on rounds 2+. Server trace middleware reads it and `core.SpanFromContext(ctx).AddLink(...)` on the dispatch span. **Star semantic** — every round 2+ links to round 1, not the previous round; backends show round 1 as the anchor regardless of MRTR depth. **Client identity capture** uses `core.WithCapturedTraceContext(ctx)` + `core.CapturedTraceContextHolder(ctx)` — the middleware writes the outbound TraceContext into the holder, the caller (CallToolWithInputs) snapshots it after round 1, stamps via `core.InjectTraceLinkIntoParams` on rounds 2+. **API gotcha**: `Client.Call` body was extracted to package-private `Client.callImpl(ctx, ...)` so MRTR can thread a holder-bearing ctx; public `Client.Call` stays ctx-free (passes `context.Background()`) for back-compat. New consumers needing ctx threading should call `callImpl` (same package only — it's package-private intentionally). Link payload is identity-only — tracestate NOT carried (link is "related-to", not full propagation). Considered alternatives both rejected: requestState-embedding (muddies opaque contract); outer-span-across-rounds (distorts duration with user-paced gather-input time). End-to-end correctness proven by `ext/otel/mrtr_tracelink_e2e_test.go` against real OTel SDK. WG-engagement note: PR 732 description has a Discord post draft asking the MRTR / Tasks WG whether to standardize the bare cross-SDK name for interop. Pattern documented in `docs/SEP_414_OTEL.md` § MRTR multi-round trace stitching + `docs/MRTR_TUTORIAL.md` Tracing section + `examples/mrtr/walkthrough.go` step.
- **ext/tasks task lifecycle spans (PR 719, issue 659)**: `tasks.Config.TracerProvider core.TracerProvider` opts the v2 runtime into SEP-414 P6 instrumentation. `task.execute` emits as a **NEW root trace** (not a child of the create span — the work outlives the few-ms `tools/call` dispatch span) via `core.WithNewRootSpan` + `core.StartSpanLinked`, carrying a Link back to the originating create span. Attributes: `mcp.task.id` at start, `mcp.task.status` at End (from final stored status). `RecordError` fires on protocol-level failures only — handler-returned errors `(nil, err)` map to v2 `completed` with `IsError=true` on the inner ToolResult, NOT `failed`. Each `tasks/get` / `tasks/update` / `tasks/cancel` dispatch span calls `core.SpanFromContext(ctx).AddLink(...)` back to the create span (the `originTC` runtime map outlives the `activeTask` entry so terminal-state polls still get the link). Sync-as-completed path (`wrapSyncAsCompletedTask`) deliberately emits NO `task.execute` — the work runs inside the create span; a zero-duration wrapper would be noise. **Contract complement landed in same PR**: `core.WithNewRootSpan(ctx) Context` + `core.IsNewRootSpanRequested(ctx) bool` — the `ext/otel` adapter's `startSpanInternal` strips any inherited OTel parent SpanContext before `tracer.Start` when the marker fires. Without this, `context.WithoutCancel` inside `core.DetachForBackground` would silently re-parent `task.execute` under the (already-ended) create span. Nil / Noop = zero overhead. ext/tasks stays dep-free of ext/otel. Pattern documented in `ext/tasks/README.md` § Tracing + `docs/SEP_414_OTEL.md` § ext/tasks task lifecycle spans + § New-root-span marker.
- **Events fanout spans (PR 730, issue 724)**: `events.Config.TracerProvider core.TracerProvider` opts every Source implementing `TracerProviderInstaller` (notably `YieldingSource`) into per-yield fanout span emission. One `events.fanout` span per emitted event (NOT per subscriber) carrying aggregate counts (`events.subscribers.total / delivered / dropped_by_match`, `events.transforms.applied`) plus `mcp.event.name` + `mcp.event.id`. Parented by the yield ctx so it stitches into the originating request trace when one exists, or starts a fresh trace from a background feeder. **Zero-subscriber yields skip emission** — idle sources would otherwise flood Tempo with empty spans every feeder tick. **Noop short-circuit** via `tp.(core.NoopTracerProvider)` type-assertion keeps the unconfigured path branch-only. Per-subscriber `events.match` / `events.transform` spans deliberately NOT shipped (would scale with subs × events, needs sampling design); the aggregate counts cover the diagnosable shape. Webhook + poll Match/Transform also NOT instrumented here (already covered by `events.webhook.deliver` from PR 714 and the events/poll dispatch span). Internal API: `YieldingSource.deliverEventToSlot` widened to return `(matched, transformed bool)`. Public `TracerProviderInstaller` interface lets TypedSource authors opt in identically. Pattern documented in `experimental/ext/events/README.md` § Tracing across the events bus + `docs/SEP_414_OTEL.md` § Events fanout span emission.
- **Events bus trace context relay (PR 712 + PR 714, issue 683)**: experimental/ext/events ships W3C Trace Context propagation across the events lifecycle. Four gates, three carriers (ctx in-process, `event.Meta.traceparent` persistent, HTTP `traceparent` header cross-process), caller-preserves rule (metaFunc-set wins). **PR 712 broke signatures** intentionally: `yield(ctx, data)` closure / `SetEmitHook(func(ctx, Event))` / `SetMetaFunc(func(ctx, Data))` / `WebhookRegistry.Deliver(ctx, event)` / package-level `Emit / EmitToWebhooks` all take ctx. **PR 714 added `WebhookRegistry.WithWebhookTracerProvider(tp)` option** emitting `events.webhook.deliver` span around each retry loop with attributes (`webhook.target.id`, `webhook.url`, `mcp.event.name`, `http.method`, `http.response.status_code`, `webhook.retry.attempts`). Nil/Noop = zero overhead. `RecordError` fires on retries-exhausted with the categorical bucket (`connection_refused` / `timeout` / `tls_error` / `http_4xx` / `http_5xx`). PR 714 also widened `Server.Broadcast(ctx, method, params)`; PR 722 (issue 715) closed the loose end — `Server.Broadcast` now injects `_meta.traceparent` via `core.InjectTraceContextIntoParams` before fan-out when ctx carries a non-zero TraceContext, so SSE-pushed notifications stitch into the originating trace. Per-transport broadcast callbacks gained ctx as a forward-compat seam (currently unused inside — the per-request `trace_middleware.WrapSessionNotifyFunc` wraps target `sc.notify`, not the transport-level base notifyFunc dispatchers expose to broadcasts). **Test sweep gotcha**: regex-based search/replace on `r.deliver(...)` call sites BREAKS when the args contain nested parens like `r.deliver(r.Targets()[0], ...)`. Use a Python paren-counter instead of regex for any mechanical signature sweep across test files. **Live verification pattern**: `curl http://localhost:3200/api/search?tags=service.name=X&limit=N` for trace IDs, then `curl http://localhost:3200/api/v2/traces/<id>` for full span tree — proves spans land in Tempo with correct attributes without needing Grafana. Adopters needing the new signatures: discord, telegram, whole-enchilada — swept in PR 714. Pattern documented in `experimental/ext/events/README.md` § Tracing across the events bus + `docs/SEP_414_OTEL.md` § Events bus trace context relay.
- **Apps Bridge trace context relay (PR 702, issue 660)**: ext/ui ships the W3C trace context relay across the iframe ↔ host postMessage boundary. TS side: `MCPApp.setTraceContextProvider(fn)` — fn returns `{traceparent?, tracestate?}` (or null to skip); merged into `params._meta` symmetrically with the MCP wire shape (uses MetaKey* names verbatim). Caller-set values preserved (mirrors Go-side `core.InjectTraceContextIntoParams`). Provider throws caught + logged. Dep-free — no OTel JS dependency. Go side: `ui.WithTracerProvider(tp)` on AppHost emits `apps.host.forward` span around `handleAppRequest`, parented by the iframe-relayed `_meta.traceparent`. Outbound `client.Call` preserves the iframe's traceparent on the wire (SEP-414 P3 caller-set-wins) so server P2 stitches the dispatch span as a child. **Test gotcha**: cross-wire trace tests MUST install `server.WithTracerProvider(...)` on the inner server — P2's middleware is what extracts `_meta.traceparent` off the wire into handler ctx. Without it, `TraceContextFromContext` on the handler returns zero even when the wire carried a traceparent. **examples/host catch-up**: PR 689's client-telemetry sweep missed `examples/host/01-apphost` + `02-multi-server` because they don't use `common.RunServer`; PR 702 caught them up. Pattern documented in `docs/APPS_DESIGN.md` § Tracing across the Apps Bridge.

- **Events spec-alignment sweep (PRs 778 / 779 / 783 / 786, spec PR1 commits 082166f0 + 99f3589c + 905ade36 + b506e347 + 21be9c31 + 6967f2ba + baef6719)** — bundled gotcha set; they bite together when touching `experimental/ext/events/`.
  - **Wire rename `params` → `arguments` was wire-breaking + Go-field-renaming (PR 778)**. JSON tag flipped on all four request structs (`events/subscribe`, `/poll`, `/unsubscribe`, `/stream`) AND the Go field renamed from `Params` → `Arguments` (compiler-driven sweep of every call site; grep-only renames silently leave sub-modules behind — see `feedback-rename-for-type-coverage` memory). Touches `RegisterParams.Arguments`, `WebhookTarget.Arguments`, `SubscribeOpts.Arguments`, Go SDK `SubscribeOptions.Arguments` / `StreamOptions.Arguments`, GORM `webhookRow.Arguments` column (renamed via fresh-deploys-only convention). Internal canonical-key computation uses the Go field — derived IDs are stable across the rename, pinned by `TestCanonicalKey_StableAcrossRename`.
  - **`WebhookTarget.ExpiresAt` is `*time.Time`, not `time.Time` (PR 779)**. Nil = no-expiry sentinel per spec PR1 commit 99f3589c §"Subscription TTL". EVERY callsite that touches expiry must guard the nil case — prune loop (`pruneExpiredLocked`), `Targets()` filter, `DeliverToTarget` liveness check, `ExpireAll` test helper. External `WebhookStore` implementors must update too. GORM column nullable (`expires_at TIMESTAMP NULL`).
  - **`ttlMs` tristate decode pattern (PR 779)** — Go's `*int64` collapses absent and JSON-null both to nil; the events/subscribe handler decodes `ttlMs` as `json.RawMessage` and pattern-matches: empty bytes → absent; literal `"null"` → no-expiry request; else parse int64. `WebhookRegistry.NegotiateExpiry(rawTTLMs json.RawMessage)` is the policy oracle (handler-private decode + clamp envelope `[MinWebhookTTL, MaxWebhookTTL]` + `WithUnsafeWebhookTTLBypass` honored + `WithAllowInfiniteWebhookTTL` gates null acceptance). Per spec there is no rejection path for TTL values — malformed input collapses silently to server default.
  - **`refreshBefore` response is `*time.Time` everywhere (PR 779)**. Always present on the wire (RFC3339 string for finite, JSON `null` for no-expiry). Go SDK `Subscription.RefreshBefore()` returns `*time.Time` — nil signals no-expiry. Refresh loop drops to 1-hour health-check cadence in that case (per spec: "clients SHOULD still re-call events/subscribe occasionally" for cursor advancement + deliveryStatus observation + reactivation).
  - **`WithAllowInfiniteWebhookTTL()` is policy, not plumbing (PR 779)**. Default OFF. Without the option, `ttlMs: null` collapses to server default. Operators flipping it ON without `WithWebhookStore(persistent)` get a stark warning at construction via `warnIfInfiniteTTLWithDefaultStore` — no-expiry subs in the in-memory store violate the spec's "MUST persist across restarts" obligation. Warning, not reject, because dev/test setups may legitimately opt in.
  - **Three distinct cleanup state machines on `WebhookRegistry` (PR 783)** — non-overlapping thresholds, non-overlapping actions:
    - TTL prune (existing, finite-TTL only — exempted for no-expiry per PR 779)
    - Suspend (existing, all subs — sliding-window consecutive failures, reversible on refresh, fires silent `terminated` envelope on transition)
    - Failure-based GC (PR 783, no-expiry only — `noExpiryFailureGCWindow` of continuous failure since last success, irreversible drop, fires PostTerminated with distinguishable message)
  - **`FailingContinuouslySince` vs `FailedSince` (PR 783)** — two anchors for two different cleanup paths. `FailedSince` resets by sliding `suspendWindow` (correct for finite-sub suspend; refresh reactivates). `FailingContinuouslySince` is **never** reset by quiet periods, only by a successful delivery (`recordDeliverySuccess` clears it). Failure-GC trigger: `target.ExpiresAt == nil && now - *FailingContinuouslySince > r.noExpiryFailureGCWindow`. `DefaultNoExpiryFailureGCWindow = 72h`; demo overrides to 2m via `EVENTS_NO_EXPIRY_GC_WINDOW`. Wire-projected as RFC3339 diagnostic (`deliveryStatusForResponse`) so subscribers can observe the GC anchor.
  - **`DeliveryStatus.Throttled` + `RetryAfterMs *int64` are projector-only (PR 778)** — wire shape lands but no in-mcpkit rate limiter sets them. Adopters wire from their own throttle state via the struct fields directly. Projected on subscribe-refresh response with omitempty; spec PR1 commit `21be9c31` distinguishes active rate-limiting from failure-driven suspension (`Active=false`).
  - **410 Gone is abandon-without-failure (PR 778)** — webhook delivery retry loop has a `case http.StatusGone:` branch BEFORE the generic 4xx catch-all. Returns without calling `recordDeliveryFailure` — `DeliveryStatus.Active` stays true, `LastError` stays `DeliveryErrorNone`, subscription untouched. Receiver said "skip THIS one," not "I'm broken." Spec PR1 commit `905ade36`.
  - **`DefaultWebhookAckTimeout = 5 * time.Second` (PR 778)** — single named constant used by both http.Client + net.Dialer. Spec PR1 commit `b506e347` says "on the order of 5 seconds is common"; mcpkit was already there. No `WithWebhookAckTimeout` option until a concrete need surfaces.

- **SEP-414 trace propagation (PR 644 contract / PR 649 server / PR 652 adapter / PR P3 client)**: `server.WithTracerProvider(...)` and `client.WithTracerProvider(...)` share the same `core.TracerProvider` contract; nil and `core.NoopTracerProvider{}` both skip the install entirely on either side — zero overhead on the unconfigured path. Both surfaces install their trace middleware OUTERMOST so user middleware (server `WithMiddleware`, client `WithClientMiddleware`) runs inside the span and contributes to recorded latency. Server inbound trace context precedence: in-band `params._meta.traceparent` wins over the `core.WithTraceContext` value on ctx (populated by the SEP-2028 HTTP `traceparent` / `tracestate` header bridge in `streamable_transport.handlePost`). Client inbound (`HandleServerRequestWithContext`) emits a wrap span around `sampling/createMessage` / `elicitation/create` / `roots/list` dispatches with the inbound `_meta.traceparent` as parent. Explicit caller-set `_meta.traceparent` on outbound messages is preserved (never clobbered, both wires). Malformed inbound traceparent drops to zero (W3C "MUST NOT forward"); dispatch still succeeds, span emits with no parent. `_meta` trace keys are bare W3C names (`traceparent` / `tracestate`), NOT under `io.modelcontextprotocol/` — that namespace is reserved for MCP-defined fields.
