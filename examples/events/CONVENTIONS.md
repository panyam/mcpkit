# `examples/events/` Conventions

Every leaf under `examples/events/` is a reference server for the [MCP Events extension](https://github.com/modelcontextprotocol/experimental-ext-triggers-events). This document pins the conventions so a reader can navigate `examples/events/<leaf>/` without re-orienting per leaf, and so new demos can be added without bikeshedding directory layout or Makefile targets.

The conventions are descriptive — `discord/` and `telegram/` already follow them. The rules below codify what's already true.

## Directory naming

- **Single-word or kebab-case.** `discord/`, `telegram/`, `kitchen-sink/`, `whole-enchilada/`, `poll-primary/`.
- **No `events-` or `mcp-` prefix on the directory** — already implied by the parent path.

## File layout — standard set

Every leaf ships the following files. Multi-binary demos (e.g., `whole-enchilada/`) have one of these layouts per binary, nested under `<binary>/`:

```
<leaf>/
├── README.md           # Quickstart + what the demo shows + setup notes
├── Makefile            # demo entry points; see "Makefile targets" below
├── main.go             # Server entry point (--serve, --tui, --note, --doc=md)
├── events.go           # EventDef + payload type registration
├── handlers.go         # MCP resource + tool handlers (if any)
├── <source>.go         # Source wiring: real upstream OR synthetic
├── walkthrough.go      # demokit step definitions
├── WALKTHROUGH.md      # Generated by `make readme` from walkthrough.go
├── e2e_test.go         # In-process e2e tests, exercise each delivery mode
├── go.mod              # Sub-module per demo
└── go.sum
```

Multi-binary demos add a `Dockerfile` and `docker-compose.yaml` at the leaf root, and place each binary under its own subdirectory:

```
whole-enchilada/
├── docker-compose.yaml
├── nginx/nginx.conf
├── event-server/{main.go, events.go, ..., Dockerfile, go.mod}
├── push-server/{main.go, ..., Dockerfile, go.mod}
├── receiver/{main.go, Dockerfile, go.mod}
└── walkthrough/{main.go, walkthrough.go, WALKTHROUGH.md, go.mod}
```

## Synthetic upstream naming

When a demo uses a synthetic event source (no real third-party integration), the file is named `synthetic_<thing>.go`. Examples: `synthetic_chat.go`, `synthetic_presence.go`. The `synthetic_` prefix is a signal to the reader that the demo is not wiring a real-world upstream — it's a generated feed for the demo's narrative.

Real-world upstreams use the upstream's name unprefixed: `discord.go`, `telegram.go`.

## Walkthrough style

- **Story-driven Q&A**, per the [discord walkthrough](discord/WALKTHROUGH.md). Each step starts with a question a host would ask: "How do I open the conversation?", "How does my agent learn what events exist?".
- **Three modes** per the demokit pattern:
  - **Live mode** (`make demo`) — interactive TUI walkthrough against a running server.
  - **Test mode** (`make demo-test`) — non-interactive run for CI / scripting.
  - **Doc mode** (`make readme`) — regenerates `WALKTHROUGH.md` from the step definitions.
- **Generated docs are checked in.** `make readme` is idempotent; a PR that touches walkthrough definitions must also commit the regenerated `WALKTHROUGH.md`.

## Makefile targets

Every leaf ships these targets. Multi-binary demos add the `demo-up` / `demo-down` cluster.

| Target | Required | Description |
|---|---|---|
| `serve` | yes | Start the server standalone (or `make demo-up` for compose-based demos). |
| `demo` | yes | Run the demokit walkthrough (interactive TUI). |
| `demo-test` | yes | Run the walkthrough in non-interactive mode for CI. |
| `readme` | yes | Regenerate `WALKTHROUGH.md` from `walkthrough.go`. |
| `build` | yes | Build the binary (or binaries, for multi-binary demos). |
| `test` | yes | `go test ./... -count=1 -v`. |
| `demo-up` | only for compose-based demos | `docker compose up -d`. |
| `demo-down` | only for compose-based demos | `docker compose down -v`. |

## Wire-shape consistency

Demos use the events library defaults. Error codes, payload shapes, and header conventions match the spec / library — never custom variants for demo convenience. If a demo needs to override a default (e.g., short TTL for the refresh-cycle step), it goes through a documented library option (`events.WithWebhookTTL`), not by patching the library or shipping a custom protocol path.

Specifically:

- **Error codes** — use spec codes: `-32011 NotFound`, `-32013 TooManySubscriptions`, `-32014 SubscriptionNotFound`, `-32015 CallbackEndpointError`. Never invent codes.
- **Webhook headers** — Standard Webhooks (`webhook-id`, `webhook-timestamp`, `webhook-signature`) by default. The `mcp` header style is available via `events.WithWebhookHeaderMode(events.HeaderModeMCP)` for compatibility with the pre-spec-r3167245184 shape.
- **Cursor shape** — pointer for cursored, `null` for cursorless. The demo never serializes `cursor: ""` to mean "no cursor."

## Mixed real-world vs synthetic upstreams

Some demos (`discord/`, `telegram/`) use real upstreams; some (`kitchen-sink/`, `whole-enchilada/`) use synthetic. The conventions are the same in both cases except the source filename (`<upstream>.go` vs `synthetic_<thing>.go`) and the walkthrough's "Setup" section (which explains either token setup or `make demo-up` orchestration).

Either is valid — neither is a "production" demo by itself. `whole-enchilada/` is the closest mcpkit ships to a "production-shape" reference (multi-tier, durable storage, multi-replica in later stages), but its synthetic upstreams keep the focus on the protocol shape rather than on integrating any particular third-party service.

## Source patterns the library offers

Demos pick one of three source patterns from `experimental/ext/events/` depending on where the events come from:

| Pattern | When | Storage owned by |
|---|---|---|
| `YieldingSource` | Source pushes events at you in-process (bot callback, channel reader, HTTP handler). | Library — bounded ring with `Recent` / `ByCursor`. |
| `TypedSource` | Source already owns its storage (DB, event log, external queue). | You — implement `Poll(cursor, limit)`. |
| `HTTPSource` | Source is a separate process pushing events over HTTP (the push-server tier in `whole-enchilada/`). | Library — wraps a YieldingSource; library mounts an `http.Handler` you attach to your mux. |

A leaf picks one or more of these and wires them per the patterns documented at `experimental/ext/events/README.md`.

## When a demo deviates

If a demo needs to violate a convention (e.g., a different Makefile target name, an unusual directory shape), the deviation must be documented in the leaf's `README.md` with a "Why this differs from CONVENTIONS.md" section. The default reader assumption is that every leaf is consistent — surprises belong in docs, not in muscle memory.
