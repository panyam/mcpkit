# mcpkit v0.4.0 (planned)

**Status: not yet tagged.** 0.4.0 is the accumulating target for a bundle of
backward-incompatible API changes. Work lands under intermediate `v0.3.x` tags
while the issues below are worked through; the `v0.4.0` tag is cut only once the
whole bundle is complete and `make audit` / `make testall` / `make testconf` are
green on a clean clone.

## Framing

0.4.0 gathers the API breaks we deliberately queued behind a version boundary
while mcpkit still has no external clients to migrate. It is an **API**-breaking
release, not a **protocol**-breaking one: no protocol capability is removed.

The SEP-2577 Roots / Sampling / Logging surfaces are **kept** in 0.4.0 with their
existing `// Deprecated:` annotations and full runtime behavior. Their removal is
deferred to a later release, no earlier than the spec itself drops them (~2027),
per [`docs/SEP_2577_DEPRECATIONS.md`](../SEP_2577_DEPRECATIONS.md). Two reasons
beyond the date gate:

1. mcpkit's own deprecation window is 12 months from the 2026-05-31 annotation
   pass (to ~2027-05-21); removing at a July-2026 tag would break that promise.
2. Those features are deprecated-but-still-in-spec on the targeted spec version,
   so conformance tests still count them. Deleting them would drop mcpkit below
   100% conformance and out of Tier-1 eligibility.

## Tier 1 follow-up (separate track, not gating 0.4.0)
Recorded so it isn't lost. Qualifying for MCP SDK **Tier 1** additionally needs:
100% conformance on the targeted spec version, a production-ready **1.0.0**
stable release (0.x doesn't satisfy the "stable release" bar), the standardized
issue labels (`bug`/`enhancement`/`question` + status + `P0..P3`), a published
dependency-update policy, and a published roadmap — then a WG advancement request.
See #431.
